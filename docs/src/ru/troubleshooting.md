# Устранение неполадок

Распространённые проблемы и их решения для ReciPro. Многие из приведённых ниже пунктов взяты из вопросов и сообщений об ошибках в [трекере проблем GitHub](https://github.com/seto77/ReciPro/issues); версия, в которой ошибка была исправлена, указана там, где это применимо.

> **Большинство проблем решается простым обновлением до [последней версии](https://github.com/seto77/ReciPro/releases/latest).** ReciPro обновляется часто, и многие из перечисленных ниже ошибок были исправлены в течение нескольких дней после их сообщения.

---

## Запуск и старт

### Симптом: процесс работает, но окно не появляется

ReciPro запускается (он виден в Диспетчере задач), но его окно так и не отображается на экране.

**Причина**: окно открылось за пределами экрана — проблема с координатами отображения Windows, обычно после смены монитора или изменения масштабирования дисплея. (Issues [#50](https://github.com/seto77/ReciPro/issues/50), [#53](https://github.com/seto77/ReciPro/issues/53), [#55](https://github.com/seto77/ReciPro/issues/55))

**Решение**:

1. Откройте **Диспетчер задач**.
2. Найдите **ReciPro** в списке процессов.
3. Щёлкните по нему правой кнопкой мыши и выберите **Развернуть**.

Окно будет перенесено на ваш основной экран. Обратите внимание, что **Переключиться на**, **На передний план** и **Свернуть** *не* помогают — работает только **Развернуть**.

### Симптом: ReciPro не запускается, аварийно завершается или зависает при старте

**Причина**: чаще всего сбой инициализации OpenGL или повреждённое значение в реестре/настройках блокирует запуск.

**Решение** (пробуйте по порядку):

1. **Отключить OpenGL**: удерживайте клавишу **Ctrl** при запуске ReciPro, чтобы стартовать с отключённым OpenGL. Недавние версии (v4.925 и более поздние) укрепляют инициализацию OpenGL, так что приложение запускается даже при сбое OpenGL — в этом случае функции 3D отключаются, но остальная часть приложения работает.
2. **Сбросить настройки**: в редакторе реестра удалите ключ `HKEY_CURRENT_USER\Software\Crystallography\ReciPro`, затем перезапустите. (Эквивалентно **Option → Reset registry**.)
3. **Чистая переустановка**: удалите ReciPro, удалите следующие папки, если они присутствуют (замените `<user>` на имя своей учётной записи), затем переустановите:
   - `C:\Users\<user>\AppData\Local\Crystallography Software\ReciPro`
   - `C:\Users\<user>\AppData\Roaming\ReciPro\ReciPro`
4. **Обновитесь** до последней версии.

Если ничто из этого не помогает, причина может быть в самой среде ОС; пожалуйста, [откройте issue](https://github.com/seto77/ReciPro/issues) с указанием сведений о вашем ПК (CPU, GPU, версия Windows).

## Проблемы OpenGL

### Симптом: чёрный экран или сбой при старте

**Причина**: несовместимый GPU или среда удалённого рабочего стола.

**Решение**:

1. Перейдите в **Option → Disable OpenGL (needs restart)** (или удерживайте **Ctrl** при запуске).
2. Перезапустите ReciPro.
3. Просмотр структуры и некоторые функции 3D будут использовать программный рендеринг.

### Симптом: встроенный или старый GPU (Intel/AMD) не выполняет рендеринг

**Причина**: некоторые встроенные GPU (например, AMD Radeon Vega, Intel UHD) имели проблемы с инициализацией OpenGL в более старых сборках. (Issue [#2](https://github.com/seto77/ReciPro/issues/2))

**Решение**: обновитесь до последней версии. Требование к версии OpenGL было понижено (v4.781), инициализация встроенных GPU была исправлена (v4.785), а сама инициализация была дополнительно укреплена для контролируемого сбоя (v4.925). Обновление драйверов вашего GPU также помогает.

### Симптом: плохое качество рендеринга

**Решение**: обновите драйверы вашего GPU. Рекомендуется внешний (дискретный) GPU с поддержкой OpenGL 1.5.

## .NET Runtime

### Симптом: приложение не запускается

**Причина**: не установлена требуемая .NET Desktop Runtime. Текущие версии требуют **.NET Desktop Runtime 10.0** (более старые сборки: v4.895–v4.91x требовали 9.0; см. issue [#43](https://github.com/seto77/ReciPro/issues/43)).

**Решение**: загрузите и установите с <https://dotnet.microsoft.com/download/dotnet/10.0> (выберите **Desktop Runtime**, x64 для большинства ПК).

### Симптом: не удаётся получить доступ к странице загрузки Microsoft

**Решение**: вы можете загрузить установщик среды выполнения напрямую. Выберите **Windows Desktop Runtime X64** для вашей архитектуры на [странице загрузки .NET 10.0](https://dotnet.microsoft.com/download/dotnet/10.0). (Issue [#49](https://github.com/seto77/ReciPro/issues/49))

## Установка

### Симптом: установка или удаление без прав администратора

**Примечание**: права администратора не требуются. Ярлыки и пользовательские файлы размещаются в ваших собственных пользовательских папках (например, `%AppData%\Microsoft\Windows\Start Menu\Programs\Crystallography Software\` и на Рабочем столе). (Issue [#38](https://github.com/seto77/ReciPro/issues/38))

## Отображение и компоновка

### Симптом: кнопки или панели обрезаны / скрыты, или компоновка выглядит сломанной

Например, кнопка **Peak Identification** в Spot ID v2 скрыта, или страница «О программе» и другие формы неправильно выровнены в недавних версиях. (Issues [#56](https://github.com/seto77/ReciPro/issues/56), [#59](https://github.com/seto77/ReciPro/issues/59))

**Причина**: регрессия масштабирования DPI / шрифта интерфейса, появившаяся в некоторых недавних сборках.

**Решение**:

- Установите **масштабирование дисплея Windows на 100 %** (это обычно восстанавливает компоновку).
- В качестве быстрого обходного решения **измените размер окна** (например, уменьшите его по вертикали), чтобы открыть скрытые элементы управления.
- Обновитесь до последней версии — компоновки исправляются постепенно. Если недавняя сборка выглядит хуже, временным вариантом является откат к чуть более старой версии (например, v4.915). Пожалуйста, сообщайте о любых остающихся сломанных формах.

## Динамические расчёты

### Симптом: очень медленно или нехватка памяти

**Причина**: слишком много блоховских волн или слишком большое изображение.

**Решение**:

- Уменьшите **No. of Bloch waves** (50–200 обычно достаточно для рутинных расчётов)
- Используйте решатель **Eigen** для ≤ 500 волн; **MKL** для > 500 волн
- Уменьшите разрешение изображения для STEM-симуляций
- Закройте другие приложения, интенсивно использующие память

### Симптом: изображение HAADF-STEM чёрное

**Причина**: атомные температурные факторы (B) установлены в ноль.

**Решение**: установите B ≥ 0.5 Å² для всех атомов. Интенсивность TDS требует ненулевых температурных факторов.

## Симулятор дифракции

### Симптом: дифракционная картина пуста / ничего не отрисовывается

**Причина**: обычно вид слишком сильно приближён, или энергия падающей волны находится вне диапазона. (Issue [#3](https://github.com/seto77/ReciPro/issues/3))

**Решение**:

- **Щёлкните левой кнопкой мыши** в основной области рисования, чтобы отдалить.
- Проверьте энергию падающей волны на вкладке **Wave** (вверху слева): рентген ≈ 1–100 keV, электрон ≈ 10–1000 keV являются подходящими.

## Ввод-вывод файлов

### Симптом: файл CIF не загружается

**Решение**:

- Проверьте, что файл CIF корректно сформирован
- Попробуйте перетащить файл на область **Crystal Information**
- Некоторые нестандартные расширения CIF могут не поддерживаться

### Симптом: файл dm3/dm4 не загружается, или "unable to cast … 'System.Single' to 'System.Double'"

**Причина**: существует несколько вариантов формата DM3/DM4, и более старые сборки не могли прочитать их все. (Issue [#15](https://github.com/seto77/ReciPro/issues/15))

**Решение**: обновитесь до последней версии — совместимость чтения DM3 была улучшена в v4.835. Если файл по-прежнему не загружается, пожалуйста, [пришлите его](https://github.com/seto77/ReciPro/issues), чтобы можно было добавить поддержку.

### Симптом: файл dm3/dm4 показывает неверный масштаб

**Решение**: проверьте калибровку в исходном программном обеспечении Digital Micrograph. ReciPro читает встроенные метаданные; если метаданные некорректны, задайте размер пикселя и длину камеры вручную на панели «Оптика».

## Часто задаваемые вопросы

### Существует ли версия для Mac (или Linux)? {#mac-linux}

Официальной версии для Mac или Linux не существует. ReciPro зависит от **.NET Desktop Runtime**, которая в настоящее время работает только под Windows. (Issue [#12](https://github.com/seto77/ReciPro/issues/12))

Тем не менее сообщалось о неофициальном способе, работающем под macOS: дистрибутив **win-x64 portable ZIP** (доступен на [странице релизов](https://github.com/seto77/ReciPro/releases/latest)) работает под macOS (Apple Silicon) с использованием оболочки Wine **Sikarugir** в сочетании с драйвером OpenGL **Mesa3D** — без лицензии Windows или виртуальной машины. Пошаговое руководство по настройке, опубликованное пользователем, доступно по адресу <https://github.com/Ryo-fkushima/ReciPro_macOS_memo>.

Обратите внимание, что эта конфигурация официально не поддерживается и не полностью проверена. Известное ограничение состоит в том, что некоторые символы (Å, надстрочные знаки, стрелки) могут отображаться неправильно.

**Исправление искажённых символов (Å, надстрочные знаки, стрелки):** причина в том, что шрифты Windows, которые ReciPro обычно использует (Segoe UI, Yu Gothic UI и т. д.), отсутствуют в среде Wine, а встроенные шрифты-заменители Wine не содержат некоторых научных глифов. ReciPro автоматически переключается на шрифты с широким покрытием, **когда обнаруживает, что работает под Wine**, поэтому исправление состоит просто в том, чтобы сделать эти шрифты доступными в Wine-префиксе:

1. Установите **DejaVu Sans** / **DejaVu Serif** (покрывают Å, надстрочные знаки, стрелки, подписи дробей) и, для японского интерфейса, **Noto Sans CJK JP** (или **Noto Sans JP**).
2. Проще всего скопировать загруженные файлы `.ttf`/`.otf` в папку шрифтов префикса — `…/drive_c/windows/Fonts/` внутри оболочки Sikarugir — затем перезапустить ReciPro. (`winetricks` также может установить некоторые из них.)
3. При перезапуске ReciPro подхватывает их автоматически; никакие настройки ReciPro менять не нужно.

Если шрифты не установлены, ReciPro сохраняет свои имена шрифтов по умолчанию, так что ничего не становится хуже — символы просто остаются искажёнными.

**Перспективы этого способа — две честные заметки:**

- Экспериментальный ZIP **win-arm64** на современных Mac **не** работает, даже на Apple Silicon: сегодняшние сборки Wine для macOS (включая Sikarugir) выполняют двоичные файлы Windows x86_64 через Rosetta 2 и не имеют механизма для запуска двоичных файлов Windows ARM64. На Mac всегда используйте portable ZIP **win-x64**.
- Apple постепенно отказывается от Rosetta 2. macOS 27 (осень 2026) анонсирована как последняя версия с полной поддержкой Rosetta 2, поэтому ожидается, что нынешний способ через x64 + Rosetta перестанет работать начиная с macOS 28 (осень 2027). Нативная ARM64-версия Wine для macOS разрабатывается в апстриме; если она появится, ZIP win-arm64 может стать преемником на Mac, но обещать этого пока нельзя.

### Работает ли ReciPro под Windows on ARM (ARM64)? {#windows-on-arm}

Да — есть два способа:

- **Нативный пакет ARM64 (экспериментальный, рекомендуемый)**: начиная с v4.938, на [странице релизов](https://github.com/seto77/ReciPro/releases/latest) публикуется экспериментальный нативный портативный пакет ARM64 (`ReciPro-v.X_arm64.zip`; до v.4.939 назывался `ReciPro-v.X-arm64.zip`). Он self-contained, поэтому установка .NET Runtime не требуется — распакуйте ZIP в папку, доступную пользователю для записи, и запустите `ReciPro.exe`. Если Windows блокирует загруженный ZIP (Mark of the Web), щёлкните по ZIP правой кнопкой мыши → **Свойства** → установите флажок **Разблокировать** → **ОК** *перед* распаковкой (или выполните `Unblock-File .\ReciPro-*arm64.zip` в PowerShell). Подробности — в прилагаемом `README-PORTABLE.txt`.
- **Пакет x64 под эмуляцией**: обычный установщик MSI и portable ZIP win-x64 также работают на ARM64-Windows через встроенную эмуляцию x64 при установленной .NET Desktop Runtime (x64) (подтверждено примерно с v4.913 с .NET 10). Тяжёлые вычисления выполняются медленнее, чем в нативной сборке. (Issue [#47](https://github.com/seto77/ReciPro/issues/47))

Примечания по нативному пакету ARM64:

- Intel MKL не существует для ARM64, поэтому соответствующие параметры решателя и пункты меню скрыты. Динамические расчёты используют прилагаемую нативную библиотеку, оптимизированную под NEON; в репрезентативных проверочных случаях её результаты совпали со сборкой x64 в пределах ожидаемой погрешности чисел с плавающей запятой.
- Виды 3D (Просмотр структуры и связанные окна) могут работать, но Windows on ARM предоставляет OpenGL только через слой трансляции Direct3D 12 (GLOn12 / Mesa), поэтому 3D-рендеринг заметно медленнее, чем на ПК с нативным драйвером OpenGL — это ограничение платформы, а не ошибка, и нативная сборка ARM64 ничего тут изменить не может. Режим прозрачности **High quality (Per-Pixel Linked List)** в Просмотре структуры особенно медленный на этом стеке драйверов; рекомендуется режим по умолчанию **Approximate**. Если виды 3D не запускаются, установите "OpenCL, OpenGL, and Vulkan Compatibility Pack" из Microsoft Store.
- Пакет ARM64 **не** работает под macOS + Wine (см. предыдущий вопрос). На Mac используйте portable ZIP win-x64.
